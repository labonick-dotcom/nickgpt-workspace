## Memory Index Failure: Complete Diagnostic

### What's actually broken

There are **two distinct failure modes** that interact. They're related but separate bugs.

---

### Failure Mode 1: Vector Store Goes `unknown` After Restart (the daily break)

**The sequence, proven from logs:**

1. Gateway runs normally. Vector store is `ready`.
2. Gateway receives SIGTERM (restart triggered -- by you, a cron, or a crash).
3. Gateway tries to drain active tasks/sessions (300s timeout). Systemd has a 30s `TimeoutStopSec`.
4. At 30s, systemd escalates to **SIGKILL**. Gateway process dies hard. Two confirmed instances:
   - Jun 11 00:49:31 → SIGKILL at 00:50:01. Memory at death: **2.4GB**.
   - Jun 12 02:04:24 → SIGKILL at 02:04:54. Memory at death: **2.9GB**.
5. New gateway process starts fresh. sqlite-vec (the native `.so` vector extension) loads, but the vector store is listed as `unknown`.
6. Any `memory_search` call now returns `"error": "index metadata is missing"`.

**Why `openclaw memory index --force` doesn't always fix it:**

The CLI command writes to the SQLite database on disk. But the running gateway holds sqlite-vec loaded in-process (in Node.js memory). These are two different things. Running the CLI rebuild doesn't guarantee the gateway's in-process state is updated. The logs show this explicitly:

```
02:02:20 → index rebuilt via CLI → gateway reports: Vector store: ready
02:04:14 → index rebuilt again  → gateway reports: Vector store: unknown
02:04:24 → SIGTERM begins
```

Running `--force` while the gateway is live can actually **flip the vector store to unknown** mid-rebuild -- the rebuild temporarily takes the in-process state offline during reinitialize, and if the gateway restarts during that window, the state is lost.

**The correct fix:** The gateway needs to either (a) auto-reinitialize its vector store on startup from the existing index without requiring a CLI trigger, or (b) expose an internal API endpoint that forces an in-process vector store reload, so the cron job can call that instead of the CLI command.

---

### Failure Mode 2: `Invalid signature in thinking block` Errors

**From logs, appearing repeatedly:**

```
messages.3.content.15: Invalid `signature` in `thinking` block
messages.7.content.100: Invalid `signature` in `thinking` block
```

Happening on Sonnet sessions (not Haiku). This fires when:
1. A session has `thinking` blocks in its transcript.
2. The gateway restarts (SIGKILL), transcript is serialized mid-session.
3. New gateway process tries to replay the transcript to Anthropic's API.
4. Anthropic rejects it: the thinking block's cryptographic `signature` is now invalid (it's tied to the original request context, not replayable).

This is causing immediate session failures after every restart -- the "interrupted session recovery" logs show it sending a failure notice to Telegram each time:
```
main-session-restart-recovery: sent interrupted main session recovery notice
main-session-restart-recovery: marked interrupted main session failed (transcript tail is not resumable)
```

This is its own bug. OpenClaw needs to strip or ignore thinking block signatures when replaying transcripts after a restart, rather than forwarding them raw to Anthropic.

### Failure Mode 3: The 4-Hour Cron Job Had a ~50% Run Failure Rate

From the 46 run history, roughly half the runs completed successfully, half failed with:
```
"⚠️ Agent couldn't generate a response. Note: some tool actions may have already been executed"

The error pattern: `input_tokens: 8, output_tokens: 59` -- extremely small. The agent fires the `exec` tool call but then fails to produce a text response after getting the result back. So the cron was unreliable as a fix even when it was running -- roughly half the runs silently failed. The vector store was only being fixed on the successful runs.

### Three distinct failure modes (all confirmed from logs)

---

**FAILURE MODE 1: Vector Store Goes `unknown` After Gateway Restart**

The gateway process holds sqlite-vec loaded in-memory (as a native Node.js extension). When the gateway restarts, this in-process state is lost. The vector store shows as `unknown`.

Running `openclaw memory index --force` from the CLI writes to disk but does NOT reliably update the gateway's in-process sqlite-vec state.

Confirmed log sequence:
```
02:02:20 → CLI index rebuild → gateway reports: Vector store: ready
02:04:14 → CLI index rebuild again → gateway reports: Vector store: unknown
02:04:24 → gateway restart begins
02:04:56 → new process starts
02:05:17 → gateway reports: Vector store: ready (after another rebuild post-restart)
```

Running `--force` while the gateway is live can FLIP the vector store to `unknown` mid-rebuild. This is a race condition.

The `memory_search` tool queries the live gateway process (not disk). So if the gateway's in-process vector store is unknown, all searches return "index metadata is missing" even though the disk index is fine.

---

**FAILURE MODE 2: Gateway Gets SIGKILL on Every Restart (Hard Kill)**

systemd `TimeoutStopSec` is 30 seconds. The gateway's drain timeout is 300 seconds. The gateway always loses the race.

Confirmed from journalctl:
```
Jun 11 00:49:31 - SIGTERM received
Jun 11 00:49:31 - draining 2 active tasks and 1 embedded run (timeout 300000ms)
Jun 11 00:50:01 - State 'stop-sigterm' timed out. Killing. [SIGKILL]
Jun 11 00:50:01 - Main process exited, code=killed, status=9/KILL
Jun 11 00:50:01 - Memory peak at kill: 2.4GB
```

Second confirmed instance:
```
Jun 12 02:04:24 - SIGTERM received
Jun 12 02:04:54 - SIGKILL
Jun 12 02:04:54 - Memory peak: 2.9GB
```

The SIGKILL means the gateway never completes a graceful shutdown, so the sqlite-vec state is never properly persisted.

---

**FAILURE MODE 3: Anthropic Thinking Block Signature Invalidation**

Every restart causes sessions with `thinking` blocks to fail because the transcript's cryptographic signatures are tied to the original request and can't be replayed after a process restart.

Confirmed log entries:
```
messages.3.content.15: Invalid `signature` in `thinking` block
messages.7.content.100: Invalid `signature` in `thinking` block
providerRuntimeFailureKind: replay_invalid
```

These errors fire immediately after restart when the gateway tries to recover interrupted sessions. They cause additional restarts, creating a loop.
